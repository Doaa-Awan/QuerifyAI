Report Column Selection — follow these rules whenever writing a SELECT statement:

1. Think about what the user needs to SEE, not just what filters to apply.
   For every table in the JOIN, ask: "Does this table contribute a column the
   user would want to read in the results?"

2. Include columns from every joined table that carry meaning for the answer.
   Example: if joining company → faults → slahead for an SLA-tickets question,
   select client name (company), SLA contract name (slahead), ticket number /
   date / title (faults).

3. Never use SELECT DISTINCT when the question implies row-level detail
   (e.g. individual tickets, invoices, calls). Use SELECT without DISTINCT and
   let the rows speak for themselves. Only use DISTINCT when the user explicitly
   asks for a deduplicated list of a single entity.

4. Follow this column priority order (left to right in SELECT):
   a. Identifier/reference columns (ticket number, invoice number, record ID)
   b. Descriptive/name columns (client name, contract name, item description)
   c. Date columns (date created, due date, date occurred)
   d. Status/type columns (status, priority, category)
   e. Numeric/amount columns (amounts, counts — only if relevant)

5. Omit columns the user did not ask about and that do not add meaning to the
   report: phone numbers, postal addresses, email addresses, internal flags,
   raw foreign-key IDs (unless they are the only identifier available).

When writing SQL queries, always alias every selected column or expression with a clear, human-readable label using AS. Rules:
- Table-qualified columns: write `table.column AS "Readable Label"` (e.g. c.cdesc AS "Client", inv.inv_amt AS "Invoice Amount")
- Aggregate functions: write COUNT(*) AS "Total", SUM(col) AS "Total Amount", AVG(col) AS "Average", MAX(col) AS "Highest", MIN(col) AS "Lowest"
- Computed expressions: alias with a descriptive label that explains the result
- Use Title Case for alias labels (e.g. "Client Name", not "client_name" or "ClientName")
- Do NOT use aliases that repeat the raw column name (e.g. avoid AS cdesc — write AS "Client Description" instead)
- If the column's meaning is ambiguous from its name, use the table description or sample data to infer a clear label
The goal is that a non-technical user can read the query results as a plain report without knowing internal column names.
