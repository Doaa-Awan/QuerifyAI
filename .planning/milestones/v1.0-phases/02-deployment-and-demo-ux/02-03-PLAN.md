<task type="checkpoint:human-action">
  <name>Task 1: Deploy to Railway and Vercel, set all env vars</name>
  <what-to-do>
Deploy both services and configure all environment variables. Follow these steps in order:

**Railway (backend):**
1. Go to railway.app → New Project → Deploy from GitHub repo → select this repo
2. Select the server as the service (or create a new service pointing to the repo)
3. In the service settings, set Root Directory to: `server`
4. Railway will detect `server/railway.toml` automatically and use it for build/start/healthcheck
5. Add environment variables (Settings → Variables). Add ALL of these:
   - `NODE_ENV` = `production`
   - `PORT` = `8080`
   - `SESSION_SECRET` = run `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"` and paste the result
   - `OPENROUTER_API_KEY` = your OpenRouter key
   - `ALLOWED_ORIGINS` = (set this AFTER Vercel deploy when you have the Vercel URL)
   - `DEMO_DB_HOST`, `DEMO_DB_PORT`, `DEMO_DB_USER`, `DEMO_DB_PASSWORD`, `DEMO_DB_NAME` = Supabase credentials
   - `DEMO_DB_SSL` = `true`
   - `DEMO_DB_HOST_SQL`, `DEMO_DB_USER_SQL`, `DEMO_DB_PASSWORD_SQL`, `DEMO_DB_NAME_SQL` = Azure SQL Server credentials
6. Trigger a deploy. Wait for it to pass the `/health` healthcheck (green in Railway dashboard).
7. Copy the Railway public URL (e.g. `https://ai-db-explorer-2026.up.railway.app`)

**Vercel (frontend):**
1. Go to vercel.com → New Project → Import Git Repository → select this repo
2. Set Root Directory to: `client`
3. Under Environment Variables, add: `VITE_API_URL` = the Railway public URL (no trailing slash)
4. Deploy. Wait for build to succeed.
5. Copy the Vercel public URL (e.g. `https://ai-db-explorer-2026.vercel.app`)

**Back to Railway:**
6. Add the remaining env var: `ALLOWED_ORIGINS` = the Vercel URL (no trailing slash)
7. Redeploy the Railway service so CORS picks up the new origin.

**Verify Railway health:**
```
curl https://<your-railway-url>/health
```
Expected: `{"status":"ok"}` with HTTP 200.

<task type="checkpoint:human-verify">
  <name>Task 2: Smoke test the live URL end-to-end</name>
  <what-built>
Railway backend with Supabase and Azure SQL Server demo DBs pre-wired. Vercel frontend with VITE_API_URL pointing to Railway. All UX components from Plans 01 and 02 are live.
  </what-built>
  <how-to-verify>
Open the live Vercel URL in a private/incognito browser window. Run these checks in order:

1. **No 404 on load:** App renders on the root URL without any errors in the browser console.

2. **SPA routing:** Navigate to the app, then manually type `/some-fake-route` in the address bar and hit enter. Should load the app (not a blank 404 page).

3. **Cold start banner (if Railway was idle):** If you have not visited the app in a while, the cold start banner should appear within 3 seconds and then disappear once the server is warm. If Railway is already warm, the banner should never appear. (To force a cold start: go to Railway dashboard, disable then re-enable the service, then immediately load the Vercel URL.)

4. **PostgreSQL demo connect:** Click the demo connect button for PostgreSQL. It should connect silently (no confirmation step). Status should show "Connected" or equivalent. The schema sidebar should populate with tables.

5. **SQL Server demo connect:** Exit/disconnect, then click the demo connect button for SQL Server. Same flow — silent connect, schema populates.

6. **Ask a question:** With the PostgreSQL demo DB connected, type a natural language question about the data (e.g. "How many rows are in each table?"). A SQL response should appear within ~15 seconds.

7. **Rate limit banner:** After asking the question, the rate limit banner should appear inside the chat panel (not on the login screen). It should show the remaining query count (likely info state — >= 10 remaining). It should NOT appear before connecting.

8. **Session persistence across refresh:** Refresh the page with the demo DB connected. The session should persist (still connected, schema still visible). If the session drops (reconnect required), note it — session persistence is dependent on Railway's proxy config working correctly.

Expected results: All 8 checks pass. The app is usable by a portfolio visitor without any local setup.
  </how-to-verify>
