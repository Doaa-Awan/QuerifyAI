In server/server.js, insert the following block immediately before the `app.listen(PORT, ...)` call (after the PORT declaration and before the commented-out async IIFE):

```js
// Startup guard — warn loudly if SESSION_SECRET is weak in production
const WEAK_SECRETS = new Set([
  'dev-secret-change-in-production',
  'change-me-to-a-long-random-string',
]);

if (process.env.NODE_ENV === 'production') {
  const secret = process.env.SESSION_SECRET || '';
  if (!secret || WEAK_SECRETS.has(secret) || secret.length < 32) {
    console.error(
      '\n[SECURITY WARNING] SESSION_SECRET is weak or missing in production.\n' +
      'Session cookies can be forged. Set a strong random secret:\n' +
      '  node -e "console.log(require(\'crypto\').randomBytes(32).toString(\'hex\'))"\n' +
      'Then set SESSION_SECRET=<that value> in your production environment.\n'
    );
  }
}
```

Placement: Insert after the `const PORT = ...` line and before the `app.listen(...)` call. Do NOT move or alter the commented-out IIFE block or the app.listen call itself.

Rationale for the three-condition check:
- `!secret` — env var missing entirely (uses code fallback which is also in the set, but belt-and-suspenders)
- `WEAK_SECRETS.has(secret)` — matches either known placeholder exactly
- `secret.length < 32` — catches short custom secrets that aren't truly random

Use `console.error` (not `console.warn`) so the message appears in stderr and is harder to miss in deployment logs. Do NOT throw — a warning allows the operator to fix it without killing an otherwise functional deployment.
