2. **`writeExplorerSnapshot` → `connectAndIntrospect`** — called internally; outer non-fatal try/catch confirmed

4. **`conversationRepository` → `chat.service.js`** — `getRecentMessages` and `appendMessage` called at lines 188, 229–230
5. **`queryCache` → `query.controller.js` + `postgresService.connect`** — `get/set/clear` all correctly wired; cache cleared on new DB connect (QAPI-03)
6. **`chatLimiter` → `POST /api/query`** — `routes.js:16` applies limiter; QAPI-02 satisfied
7. **`ChatBot` → `DbExplorer.jsx`** — `onTablesUsed` prop chain fully wired to `highlightedTables` CSS class
8. **Phase 01 → Phase 04 implicit dependency** — `writeExplorerSnapshot` produces `table-metadata.json`; `loadTableMetadata` in `chat.service.js:34` reads it for two-pass table selection; graceful degradation to full-schema confirmed

### Issues Found (2/10)

**Issue 1 — Orphaned route `POST /api/connect` (affects future SEC-01)**
`routes.js:20` defines this endpoint; `postgresController.connectAndIntrospect` sets `req.session.connected = true`. The client never calls this route — it uses `POST /db/connect` which does NOT set the session flag. Net effect: `req.session.connected` is never `true` in normal user flow. Adding `requireSession` to query endpoints (SEC-01) will break all users until this is fixed.

**Issue 2 — Inconsistent API response casing**
`POST /api/chat` returns `tables_used` (snake_case). `POST /api/query` returns `tablesUsed` (camelCase via remap in `query.controller.js:61`). Not a current regression but a footgun for any direct API consumer.
